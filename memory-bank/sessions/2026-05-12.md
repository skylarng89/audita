# Session Log — 2026-05-12

## Session Hardening Completion & SecurityConfig Stabilization

### What Was Done

1. Completed security-first session hardening end to end
   - Fixed refresh-cookie scope so `/api/v1/auth/logout` receives the refresh cookie and can revoke backend refresh state.
   - Removed JS-readable auth-token persistence from the frontend store.
   - Added non-rotating `POST /api/v1/auth/session` for cold-start restore from the HttpOnly refresh cookie.
   - Restricted silent refresh to `401` responses only.
   - Added frontend/backend API contract enforcement via `X-Audita-Api-Contract`.
   - Added token-free cross-tab restore/logout sync via `BroadcastChannel` and `localStorage` fallback.

2. Added focused auth/session regression coverage
   - Backend: `AuthControllerWebMvcTest`, `ApiContractHeaderFilterTest`, `TenantResolutionFilterTest`, `AuthServiceTest`.
   - Frontend: `tests/auth/session.spec.ts`, `tenant-resolution.spec.ts`, `api-contract.spec.ts`, `session-sync.spec.ts`, `tests/middleware/tenant.spec.ts`, `tests/middleware/auth.global.spec.ts`.

3. Replaced the temporary Spring Security workaround with a long-term public API fix
   - Removed the reflection-based authorization-rule bridge from `SecurityConfig`.
   - Rebuilt request authorization with `RequestMatcherDelegatingAuthorizationManager`, `AuthorizationFilter`, and `PathPatternRequestMatcher`.
   - Added `SecurityConfigAuthorizationTest` to lock public auth routes, authenticated fallback, and super-admin platform boundaries.

### Outcome

- Session recovery now either succeeds through the existing secure refresh-cookie path or fails closed without stale half-authenticated UI state.
- Logout can revoke refresh state reliably.
- The auth/session layer no longer depends on JS-readable token persistence.
- The Spring Security configuration no longer depends on reflection or the editor-hostile nested DSL type.
