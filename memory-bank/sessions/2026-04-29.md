# Session Log — 2026-04-29

## Post-Sprint UX Hardening: Browser Bootstrap 403 Root Cause and Fix

### What Was Done

1. Reproduced persistent browser failure
   - Browser submit on `/platform/bootstrap` consistently returned 403 with UI fallback banner.
   - Immediate CLI POST to the same proxied endpoint often returned 200, confirming browser-vs-CLI mismatch.

2. Hardened frontend bootstrap request path
   - Updated bootstrap endpoint detection in `audita-web/plugins/api.ts` to inspect `request` argument.
   - Ensured bootstrap calls use anonymous header behavior and omit credentials where appropriate.
   - Added explicit `credentials: "omit"` for onboarding status fetch and bootstrap submit.

3. Added backend diagnostics
   - Added focused bootstrap-path diagnostics in:
     - `JwtAuthenticationFilter`
     - `TenantResolutionFilter`
     - `PlatformBootstrapController`
     - `GlobalExceptionHandler`
   - Rebuilt API and collected targeted logs during repro attempts.

4. Confirmed true failure mode
   - Browser-side fetch to `/api/platform/v1/bootstrap` returned:
     - status: 403
     - body: `Invalid CORS request`
   - API logs showed status GET handling but no bootstrap POST controller path during failures, indicating rejection before business logic.

5. Implemented final fix
   - Updated `audita-web/server/routes/api/[...path].ts` internal proxy route to remove:
     - `origin`
     - `referer`
     - `host`
   - Rebuilt web service and re-tested browser path.

### Outcome

- Browser fetch POST to `/api/platform/v1/bootstrap` now returns 200 with success payload.
- Onboarding state transitions to `onboardingCompleted=true`.
- Reload from bootstrap route redirects to `/auth/sign-in` as expected.

### Follow-Up

- Add an automated regression test for the Nuxt internal proxy bootstrap path to detect future CORS/header regressions early.
