## Sprint 7 ŌĆö File Security Hardening, Custom Fields UX Redesign & CR Edit Mode

### What Was Done

#### 1. File Upload Security Hardening (3-Layer Validation)

**Problem**: Attachment upload had no type enforcement beyond file extension.

**Solution**: Three independent layers now gate every upload.

- **Layer 1 ŌĆö Config (`.env` / `@Value`)**: `UPLOAD_ALLOWED_MIME_TYPES` (comma-delimited), `UPLOAD_MAX_SIZE_BYTES`, `UPLOAD_MAX_SIZE`. Defaults baked into `@Value` fallback.
- **Layer 2 ŌĆö Backend (`ChangeRequestService`)**: Three new methods:
  - `isAllowedMimeType(String mimeType)`: checks against the config-driven allowlist.
  - `isExtensionAllowed(String fileName, String mimeType)`: switch on MIME type ŌåÆ expected extension(s). Handles DOCX/XLSX ambiguity (both are ZIP/PK, disambiguated by extension).
  - `isSignatureValid(MultipartFile, String mimeType)`: magic byte checks ŌĆö `%PDF-` for PDF, `89 50 4E 47` for PNG, `FF D8 FF` for JPEG, `50 4B 03 04` for DOCX/XLSX.
  - Path traversal guard: `outputPath.normalize()` then `outputPath.startsWith(storageDir)` assertion.
- **Layer 3 ŌĆö Frontend (`[id].vue`)**: `ALLOWED_EXTENSIONS` and `ALLOWED_MIME_TYPES` sets + `isFileTypeAllowed(file)` function validates both extension and MIME before dispatching `uploadAttachment()`.

**Files Modified**:

- `audita-api/infrastructure/src/main/java/io/audita/infrastructure/service/ChangeRequestService.java`
- `audita-api/.env`
- `audita-web/pages/change-requests/[id].vue`

#### 2. Filename Normalization

**Problem**: Uploaded files preserved original names including spaces, special chars, mixed case ŌĆö unsafe for filesystem paths.

**Solution**: `normalizeFileName(String original)` helper in `ChangeRequestService`:

- Lowercases stem and extension.
- Replaces runs of non-alphanumeric chars with a single hyphen.
- Strips leading/trailing hyphens.
- Example: `"Network Diagram - Acme Inc.DOCX"` ŌåÆ `"network-diagram-acme-inc.docx"`.
- Original display name preserved in DB (`file_name` column); stored path uses normalized name.

#### 3. Custom Fields 400 Fix

**Root cause**: `addCustomField()` pushed `{ fieldId: "", value: "" }`. Backend `@NotNull UUID fieldId` cannot deserialize an empty string ŌåÆ `400 Bad Request`.

**Fix**: Removed the "Add Row" pattern entirely. Custom fields are now driven by field definitions: one row rendered per definition, always with a valid UUID key. The old form-driven approach was replaced with a definition-driven read pattern.

## Sprint 7 (Continued) ŌĆö VueDatePicker ŌåÆ Native Date/Time Inputs

### Problem

VueDatePicker stored a full `Date` object internally regardless of `model-type="yyyy-MM-dd"`. When bound to a date-only field, the time component leaked into the display. Multiple approaches failed (combined flow, split pickers, `:format` prop, `model-type` string). Root cause: VueDatePicker's display logic always processes the full `Date` including time ŌĆö no prop prevents it.

### Solution

Replaced all 4 VueDatePicker instances in both Create CR and CR Edit pages with native `<input type="date">` + `<input type="time">`. Native inputs are architecturally incapable of mixing concerns. Dark mode: `:style="{ colorScheme: isDark ? 'dark' : 'light' }"` on each input.

### Changes Applied

1. **`audita-web/pages/change-requests/new.vue`**
   - 4 `<ClientOnly><VueDatePicker .../>` ŌåÆ 4 native inputs with `colorScheme` style binding
   - State: `null as { hours: number; minutes: number } | null` ŌåÆ `""` for all 4 fields
   - `combineParts(dateStr, time: {...})` ŌåÆ `combineParts(dateStr: string, timeStr: string)`
   - `onStartChange` clears `scheduledEndTime = ""` (was `null`)

2. **`audita-web/pages/change-requests/[id].vue`**
   - Same template + type + `combineParts` changes
   - `enterEditMode` now serialises times as `"HH:mm"` template literals (was `{ hours, minutes }` object)
   - All clears use `""` instead of `null`

3. **`audita-web/plugins/vue-datepicker.client.ts`** ŌĆö deleted

4. **`audita-web/assets/css/main.css`** ŌĆö removed entire `.dp__*` VueDatePicker CSS block (~50 lines)
