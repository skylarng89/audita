## Changes

### Setup Token Injection
- **Root cause**: `X-Setup-Token` header not sent by frontend, backend rejected onboarding
- **Fix**: Nuxt proxy handler (`server/routes/api/[...path].ts`) injects `X-Setup-Token` from server-side `NUXT_SETUP_TOKEN` env var; `nuxt.config.ts` adds `setupToken` runtime config; compose files pass `NUXT_SETUP_TOKEN: ${BOOTSTRAP_SETUP_TOKEN}`

### SLA Component Silent Failure (ADR-035 pattern)
- **Root cause**: `SettingsSla.vue` SFC with complex prop types silently failed in Nuxt 4 SPA mode â€” same generic SFC issue as ADR-035/036
- **Fix**: Replaced `<AdminSettingsSla>` component with raw inline `<input v-model.number>` bindings in `pages/admin/settings/index.vue`; deleted `components/admin/SettingsSla.vue`

### Loading Overlay â†’ Skeleton Loaders
- **Root cause**: Global overlay had race conditions, pages without `hideLoading()` calls stuck forever, logout broke it
- **Fix**: Created `SharedFieldSkeleton.vue` (`animate-pulse`), added inline skeletons to all 20 pages, removed `useLoadingOverlay.ts`, `SharedLoadingOverlay.vue`, all overlay wiring in `app.vue`, all `hideLoading()`/`showLoading()` calls

### Log Permissions (Container User Override)
- **Root cause**: Hardened image runs as `USER 65532`, bind-mount directories owned by root â†’ `Permission denied`
- **Fix**: Changed log path to `/tmp/logs/audita-api.log` (world-writable), added `user: "${APP_UID:-1000}"` to both compose files, bind mount `./logs:/tmp/logs`

### LazyInitializationException in Watcher List
- **Root cause**: `CrWatcherRepository.findByChangeRequestId()` + `RequestUatWatcherRepository.findByUatId()` didn't eagerly fetch `user` relationship. DTO mapping accessed `getUser().getEmail()` outside transaction (OSIV disabled)
- **Fix**: Added `@EntityGraph(attributePaths = {"user"})` to both repository queries (matching `CrApproverRepository` pattern)

### Audit Log UPDATE Instead of INSERT
- **Root cause**: `AuditLogRepository.findTopByOrderByChainIndexDesc()` had `@Lock(PESSIMISTIC_WRITE)`. When `addWatchers()` called `auditLogService.log()` in a loop, the second call's `REQUIRES_NEW` transaction acquired a write lock on the row just inserted, Hibernate dirty-checked the locked entity â†’ UPDATE â†’ append-only trigger rejected it
- **Fix**: Replaced `@Lock` entity query with native `getMaxChainIndex()` + `findLastRecordHash()` native queries. Replaced `auditLogRepository.save()` with native `INSERT INTO audit_log` via `EntityManager` â€” zero entities loaded into persistence context, pure INSERT

### Approver Not Removed on Demotion
- **Root cause**: `moveApproverToWatcher` called `crApproverRepository.delete(approver)` without removing from in-memory `cr.getApprovers()` collection. With `@OneToMany(orphanRemoval=true, cascade=ALL)`, Hibernate re-persisted the deleted entity at flush
- **Fix**: Added `cr.getApprovers().remove(approver)` before `crApproverRepository.delete(approver)` in `moveApproverToWatcher` and `removeApprover`

### Activity Stream Missing for Watcher/Move Actions
- **Root cause**: 4 watcher/move methods in `ChangeRequestService` called `auditLogService.log()` but never `logActivity()`. `RequestUatService` counterparts did both
- **Fix**: Added `logActivity()` calls for `CR_WATCHER_ADDED/REMOVED/PROMOTED`, `CR_APPROVER_DEMOTED`

### Mark Complete â€” Restrict to Creator + Admin
- **Root cause**: Frontend `CrCompletionStatusControl` showed button to anyone with `cr.manage_participants`, no identity check. Backend correctly restricted via `assertCanMutate` but button should mirror it
- **Fix**: Added `isCreator` prop to `CrCompletionStatusControl.vue`, button gates on `(isCreator || isAdmin || isSuperAdmin)`, passed `:is-creator="isCreator"` from detail page

### Comment Mentions Broken
- **Root cause**: Frontend sent `q=<query>` but backend expected `query=<query>`. Empty query â†’ all users returned. Admin users appeared first due to UUID ordering
- **Fix**: Changed parameter name to `query`, switched to dedicated `GET /api/v1/users/mention-candidates` endpoint (filters by ACTIVE status). Added controller mapping for existing `UserService.searchMentionCandidates()`

### File Upload Permission Error
- **Root cause**: `user:` override in compose runs container as uid 1000, but named volume `api_uploads:/data/uploads` (then `/tmp/uploads`) retained root ownership from previous runs. `chmod 777` in Dockerfile didn't help because named volumes persist independently
- **Fix**: Removed named volume entirely. Changed to bind mount `./uploads:/tmp/uploads` (same pattern as logs). Changed default `STORAGE_LOCAL_BASE_PATH` to `/tmp/uploads`. Cleaned up Dockerfile. User must `mkdir -p uploads && chmod 777 uploads` on host

### Multi-File Upload on Detail Page
- **Root cause**: `[id].vue` missing `multiple` attribute on file input, handlers read only `.item(0)` instead of iterating all files. `new.vue` already handled multi-file correctly
- **Fix**: Added `multiple` attribute, rewrote `onSelectUpload` and `onDropUpload` to loop over `files`, updated text to "Select Files"
