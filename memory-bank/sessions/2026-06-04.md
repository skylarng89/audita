## Session Focus

Sprint 15 — Requests Workflow Expansion: full design, planning, and implementation via subagent-driven development.

## Work Completed

### Product and Architecture Clarification

- Captured and resolved workflow ambiguity with explicit rules for two modes:
  - `APPROVAL_ONLY`
  - `DELIVERY_PIPELINE` (`Request -> UAT -> Deployment`)
- Confirmed delivery governance rules:
  - UAT actions allowed only after main request approval status is `APPROVED`.
  - UAT promotion requires required approvers only.
  - UAT becomes read-only after promotion but still accepts comments.
  - Deployment creation is promotion-only; no direct creation.
  - Completion status update authority limited to creator/Admin/Super Admin.

## Decisions Made

- Keep compatibility mode for API/routes (`/change-requests`) in this release.
- Rename user-visible labels/navigation to "Requests".
- Use immutable persisted display IDs (`<PREFIX>-<zero-padded-sequence>`) with admin-configurable prefix for future records only.
- Use department master data from Admin Settings as source for request department dropdowns.
- Require activity stream and audit trail parity for every UAT/deployment action.

### Code Quality Fixes Applied During Review
- **RW15-001**: Dual-status sync gap — `submit()`, `cancel()`, `evaluateApprovalClosure()` now sync `approvalStatus` with `status`.
- **RW15-005**: NPE guard on null/blank org setting value in `generateDisplayId()`.
- **RW15-006**: Added audit logging to `completeRequest()` and `setWorkflowMode()`; null-check on `mode` parameter.
- **RW15-007**: Added `readOnly` guard on `approveUat()`/`rejectUat()`; vacuous-truth fix requiring at least one required approver before promotion.
- **RW15-008**: Added deployment status guards on `approveDeployment()`/`rejectDeployment()` — must be `PENDING_APPROVAL`.

### Key Architecture Decisions
- **Conditional workflow model** over universal UAT/Deployment — avoids forcing technical workflow on non-technical departments.
- **Compatibility mode** — routes stay as `/change-requests`, only UI labels renamed to "Requests".
- **Separate stage entities** (`request_uat`, `request_deployments`) over polymorphic stage engine — simpler, stronger constraints, easier audit.
- **Immutable display IDs** — prefix changes only affect future records; existing IDs never rewritten.
- **Bidirectional links** with canonical UUID ordering — `request_id_a < request_id_b` enforced at DB and app layer.
